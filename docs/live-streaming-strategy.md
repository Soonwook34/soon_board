# OpenF1 라이브 스트리밍 운영 전략

> 정리일: 2026-05-20

## 2. 핵심 아이디어: 데이터 시계 기반 30초 플레이아웃 버퍼

비디오 스트리밍의 **jitter buffer / playout buffer** 패턴을 그대로 적용한다. OpenF1의 모든 레코드가 절대 UTC timestamp(`date`)를 들고 있다는 점이 이 패턴을 가능하게 한다.

```
                     ┌── 데이터 시계 (fetch edge) ── newest_date ≈ wall_clock − 3s
                     │
                     │      ◄────── 버퍼 (목표 30초) ──────►
                     │
표시 시계 (display) ◄┘   display_clock = newest_received_date − 30s
```

### 2.1 표시 시계 정의
- **`display_clock = newest_received_date − 30s`**
  - 클라이언트 wall clock이 아니라 **데이터 시계**로부터 파생.
  - 데이터가 늦게 오면 표시도 자연스럽게 같이 늦어진다 → "끊김"이 아니라 "느려짐"으로 나타남.
  - 사용자 기기 시계 오차/드리프트의 영향을 받지 않는다.

### 2.2 렌더 루프
- 표시 엔진은 고정 fps(예: 30 fps)로 `display_clock`을 진행시킨다.
- 매 프레임마다 차량별로 "`display_clock` 시점의 위치"를 보간으로 산출해 그린다.
- 보간은 **차량별 독립 시계**로 한다 (`location`의 timestamp가 차량마다 살짝 어긋나므로 전체 시간축에 한꺼번에 정렬하지 말 것).

### 2.3 버퍼 운영 규칙
| 버퍼 깊이 | 동작 |
|---|---|
| **20~30초** | 정상. UI 인디케이터: "라이브 (30s 지연)". |
| **10~20초** | 경고. UI 인디케이터: "데이터 수신 지연". 표시 시계 전진 속도 그대로. |
| **5~10초** | 위험. 표시 시계 전진 속도를 살짝 늦춰(예: 0.9x) 버퍼 회복 유도. |
| **< 5초** | 표시 시계 일시정지. UI: "데이터 수신 중단". 데이터 회복 시 자동 재개. |

## 3. 폴링 예산

### 3.1 무료 플랜 (한도 30 req/min, 3 req/s)

| 엔드포인트 | 폴링 주기 | req/min | 폴링 1회 응답 (참고) | 메모 |
|---|---|---|---|---|
| `location` | 10s | 6 | ~740 records (10s × 3.7 Hz × 20명) | 버퍼 항상 20~30s 유지 |
| `position` | 10s | 6 | 가변 (이벤트성) | 순위 |
| `race_control` | 15s | 4 | 보통 0~2건 | 깃발/SC |
| `intervals` | 15s | 4 | ~80 records | 갭 (derive 없이 직접 폴) |
| `laps` | 30s | 2 | 랩 완료 시점 신규 | |
| `pit` | 30s | 2 | 보통 0건 | |
| `stints` | 60s | 1 | 거의 0건 | |
| `weather` | 60s | 1 | 1건 | |
| **합계** | | **26 req/min** | | **429 대비 여유 4 req/min** |

- 모든 엔드포인트가 같은 ~30s 지연으로 표시됨 (일관된 UX).
- freshness 우선순위가 사라지므로 적응형 cadence/이벤트 트리거가 불필요 — 로직이 단순해진다.

### 3.2 유료 (스폰서) 플랜

| 엔드포인트 | 전송 | 비고 |
|---|---|---|
| 위 8개 엔드포인트 전부 | **MQTT 구독** (`mqtts://mqtt.openf1.org:8883`) | REST 폴링 0 |
| REST 60 req/min 예산 | hydration, 백필, 끊김 복구 | 사실상 미사용 |

- MQTT 푸시가 sample을 그대로 흘려보내므로 버퍼는 더 빠르고 균일하게 채워진다.
- **30초 버퍼는 그대로 유지**. 유료라도 OpenF1 자체 장애(2026-03 race day 다운 등)에 대비한 안전망은 동일하게 필요.

## 4. 채택한 운영 트릭 (안정성 검증된 것만)

| 트릭 | 채택 | 이유 |
|---|---|---|
| 정적 데이터(`meetings`/`sessions`/`drivers`/서킷 transform) 1회 캐시 | **O** | 변동 없음 |
| `date>=cursor` 커서 폴링 (이미 받은 데이터 재요청 금지) | **O** | REST 기본 위생, 위험 0 |
| `session_key=latest` + 오프시즌 404 핸들링 | **O** | 검증된 동작, 사전 lookup 1회 절감 |
| 백엔드 1회 폴 → 자체 WebSocket으로 클라이언트 팬아웃 | **O** (필수) | 동접 N명일 때 OpenF1 부하 N배 폭증 방지 |

## 5. 의도적으로 채택하지 않은 트릭

| 트릭 | 비채택 사유 |
|---|---|
| `intervals` derive (position+laps에서 계산) | 의미가 다름(랩 단위 snap). 디버깅 비용 증가. 4 req/min 절약은 30s 버퍼 운영의 안정성을 해칠 가치가 없음 |
| `stints` derive (pit+laps에서 계산) | OpenF1 Issue #89(스틴트 경계 랩 중복)에 직접 노출. 자체 계산으로 더 불안해질 수 있음 |
| 이벤트 트리거 폴링 (pit 감지 시 stints 폴 등) | 엔드포인트 간 의존이 생겨 한쪽 늦으면 다른 쪽이 멈춤 |
| 적응형 cadence (SC 중엔 절감, 재시작 시 부스트) | 분기 많고 테스트 어려움. 30s 버퍼면 freshness 우선순위가 균등화되어 불필요 |
| 랩 종료 감지로 laps/pit/stints coalesce | 위와 동일. 일관 cadence가 더 단순하고 안정 |

## 6. 워밍업 (페이지 첫 로드 시)

**백엔드 팬아웃 구조에서 워밍업은 자동으로 해결된다.**

- 백엔드 프로세스는 항상 최근 30s 데이터를 버퍼에 보유한다.
- 새 사용자가 접속하면 백엔드가 보유한 **최신 30s 버퍼를 1회 스냅샷으로 전송**, 이후엔 WebSocket push만 이어간다.
- 사용자는 "30초 카운트다운" 없이 즉시 영상이 흐르는 듯한 경험을 얻는다.

**백엔드 없이 클라이언트 직접 폴링하는 경우 (권장하지 않음):**
- 페이지 로드 → 즉시 `date>=now-35s & date<=now-3s` 범위로 hydration 1회.
- 그 응답으로 버퍼를 채운 뒤 정상 폴링 루프 진입.
- 매 사용자가 자기 버퍼를 새로 만들어야 하므로 OpenF1 부하가 사용자 수에 비례.

## 7. 운영 시나리오별 흡수 결과

| 상황 | 일반 폴링 운영 | 본 전략 (30s 버퍼) |
|---|---|---|
| 네트워크 지터 1~3s | 표시 jerky | **무영향** (버퍼가 흡수) |
| 429 백오프 5~10s | 표시 중단/jump | **무영향** (버퍼 잠시 얕아짐) |
| OpenF1 hiccup 20s | 표시 멈춤 | **거의 무영향** (버퍼 회복) |
| OpenF1 다운 30s+ | 표시 멈춤 | 30s 시점부터 "데이터 지연" 뱃지, 표시 시계 천천히 진행 |
| OpenF1 장기 다운 (Miami 2026 사례) | 즉시 사용 불가 | 30s 후 장애 알림, 사용자가 명시적으로 인지 |
| 세션 종료 후 백필 | 별도 처리 코드 필요 | **동일 버퍼 메커니즘으로 자연 통합** (재생 모드와 코드 동형) |

## 8. 데이터 모델·코드 설계 가이드

### 8.1 백엔드 컴포넌트
1. **Poller** — OpenF1 REST를 §3.1 cadence로 폴링. 각 엔드포인트별로 `last_cursor`(가장 최신 받은 레코드의 `date`) 관리.
2. **Buffer (in-memory + Redis)** — 엔드포인트별 최근 60초 분량의 레코드 ring buffer. 30s 표시 + 30s 안전 마진.
3. **Fan-out** — 자체 WebSocket 서버. 신규 접속 시 버퍼 스냅샷 전송, 이후엔 Poller 신규 레코드를 모든 구독자에게 push.
4. **Health monitor** — 버퍼 깊이, 429 카운트, 최신 `date` 추세를 메트릭으로 노출.

### 8.2 무료 → 유료 전환 인터페이스
- Poller를 추상화: `DataSource` 인터페이스로 감싼다.
  - `RestPollingSource` — 무료 플랜용 구현.
  - `MqttSubscriptionSource` — 유료용 구현.
- Buffer/Fan-out/Health monitor는 데이터 소스에 의존하지 않는다.
- 전환 비용 = 어댑터 한 클래스 교체.

### 8.3 클라이언트 컴포넌트
1. **WebSocket client** — 백엔드 자체 WebSocket에 접속. 스냅샷 수신 후 푸시 구독.
2. **Buffer** — 엔드포인트별 시계열 (특히 `location`은 차량별 별도 시계열).
3. **Display clock** — `newest_received_date − 30s`.
4. **Render loop** — 30 fps. 매 프레임마다 표시 시계 시점의 차량 위치를 차량별 보간으로 계산해 그림.
5. **Status indicator** — 버퍼 깊이 기반(§2.3) UI 상태 머신.

### 8.4 보간 (interpolation) 권장 방식
- `location`은 ~3.7 Hz × 20명. 차량별로 시계열을 분리.
- 차량별 인접 두 sample 사이를 **시간 가중 선형 보간** (낮은 비용, 시각적으로 충분히 매끈).
- 5초 이상 sample이 빠진 차량은 보간 대신 마지막 위치를 깜빡임/투명도 변화로 표시 ("연결 불안정") — 빈 곳을 직선으로 가로지르는 부자연스러운 움직임 방지.

## 9. 운영 지표 (모니터링 대상)

| 지표 | 정상 범위 | 알림 임계 |
|---|---|---|
| 평균 버퍼 깊이 | 25~30s | < 15s 5분 지속 |
| 표시 시계 vs wall clock 격차 | 30~33s | > 60s |
| 429 응답 비율 (분당) | 0% | > 0% |
| `location` 폴 1회 응답 시간 | < 500 ms | > 2 s |
| `newest_date` 진행률 (wall clock 1초당 진행 초) | ~1.0 | < 0.5 5분 지속 (OpenF1 정체 의심) |
| 클라이언트 WebSocket 평균 연결 시간 | 세션 시간 ≈ | 잦은 재연결 (네트워크/백엔드 이슈) |

## 10. 주의 사항

1. **알림성 정보도 30초 지연됨** — "DRS 오픈", "옐로우 깃발" 등이 실제 사건 30초 후 표시됨. 시청 UX엔 무관, 베팅·실시간 의사결정 UX엔 부적합.
2. **타임존** — 표시 시계의 기준은 항상 UTC `date`. UI에서 사용자 로컬 시간으로 보여줄 때만 변환.
3. **세션 경계** — 첫 30초는 버퍼가 미완성, 세션 종료 후에도 30초간 마저 재생됨. 두 경계에서 UI 메시지(`곧 시작합니다`, `세션 종료, 마저 재생 중`)로 안내.
4. **차량 timestamp 어긋남** — `location`은 차량별 sample timestamp가 미세하게 다름. 차량별 독립 시계로 보간할 것.
5. **`location` 가라지 sentinel** — (0, 0, 0) 부근 값은 가라지/핏 정지 상태일 수 있음. 실주행 좌표로 간주하지 말 것.
6. **`intervals` 문자열 갭** — 랩 다운된 차량은 `"+1 LAP"` 같은 문자열로 옴. 숫자 연산 전 타입 체크 필수.
7. **백엔드 팬아웃 미적용 시 무료 플랜 즉시 폭파** — 사용자 N명이 각자 클라이언트에서 OpenF1를 직접 폴링하면 N × 26 req/min. 동접 2명만 되어도 한도 초과. 팬아웃은 권장이 아니라 **사실상 필수**.
8. **`csv=true` 옵션 사용 금지** — 본 운영에선 JSON만 사용. CSV는 파싱 비용·메모리 사용량이 더 크다.
9. **OpenF1는 SLA가 없는 단일 메인테이너 프로젝트** — 본 전략은 흡수 가능한 장애 윈도우를 ~30s까지 확장할 뿐, 더 긴 장애에 대한 보장은 아니다. 장기 다운에 대한 UX는 별도 fallback(예: 정적 결과 표시)로 정의해 둘 것.

## 11. 미해결 / 후속 결정 필요 항목

- 세션 종료 직후 OpenF1가 historical 보존을 시작할 때, 라이브 버퍼에서 historical 쿼리로 전환하는 분기 로직.
- `intervals`가 historical에서 비신뢰임을 감안할 때, 본 라이브 화면이 그대로 "직후 다시보기" 모드까지 커버할지 (커버한다면 `intervals`만은 라이브 중 별도 영구 저장).
- 백엔드 다중 인스턴스 운영 시 polling 중복 방지 (리더 선출 / 단일 polling 인스턴스).
- 클라이언트 WebSocket 재접속 시 hydration 응답 크기 상한 (장시간 재접속 회피).
