# OpenF1 과거 경기 재생(Replay) 운영 전략

> 정리일: 2026-05-20
> 전제: 본 문서는 종료된 F1 세션을 사용자가 재생/스크럽/배속 변경하며 시청할 때의 OpenF1 데이터 소비 전략을 정의한다. API 자체에 대한 사실은 [openf1-api-reference.md](./openf1-api-reference.md), 라이브 모드 전략은 [live-streaming-strategy.md](./live-streaming-strategy.md)를 따른다.

> **본 MVP 운영 컨텍스트:** GitHub + Vercel 정적 호스팅 + 개인 사용(동접 1명). 폴러는 **사용자 브라우저**가 담당하고, 윈도우 캐시는 **세션 동안 메모리에만** 유지된다(탭 닫으면 소실, 재방문 시 재폴). 다중 사용자 LRU 캐시·request coalescing·핫 세션 사전 적재는 외부 공개 시점 확장 참고용으로만 남긴다 (§5, §10-9 참고).

## 0. 한눈에 보는 운영 사양

| 항목 | 값 |
|---|---|
| **데이터 수집 패턴** | **반-개구간 윈도우 폴링** `date>=T & date<T+W` |
| 폴러 위치 | **사용자 브라우저** (백엔드 없음) |
| 캐시 위치 / 수명 | **브라우저 메모리만** — 탭 닫으면 소실 |
| 윈도우 크기 (조밀 엔드포인트) | 60s (`location`, `position`, `intervals`) |
| 윈도우 크기 (희박 엔드포인트) | 세션 전체 1회 (`laps`, `weather`, `race_control`, `pit`, `stints`) |
| 재생 버퍼 깊이 (1x 속도) | 60s 룩어헤드 |
| 재생 시작까지 워밍업 | ~1~2초 (희박 엔드포인트 풀세션 + 최초 윈도우 1개) |
| 사용 가능 데이터 시작점 | **2023 시즌** (전 엔드포인트 공통) |
| 인증 | 불필요 (historical은 무료/익명) |
| 평균 요청 부하 | **약 3 req/min** (1x 재생) / **약 12 req/min** (4x 재생) |
| `intervals` historical 가용성 | **불완전** ("live only"로 기술됨, §10 대응책 참조) |

## 1. 스코프와 라이브와의 차이

### 1.1 동일하게 표시할 엔드포인트
[live-streaming-strategy.md §1.1](./live-streaming-strategy.md)과 동일.
- 표시 포함: `location`, `position`, `intervals`, `race_control`, `laps`, `pit`, `stints`, `weather`
- 표시 제외: `car_data`, `team_radio`

### 1.2 라이브와의 핵심 차이
| 항목 | 라이브 모드 | 재생 모드 |
|---|---|---|
| 데이터 도착 시간 모델 | 약 3s 업스트림 지연 + 폴링 간격 | 즉시 (이미 보존된 데이터) |
| 재생 시계 | `newest_received_date − 30s` (데이터 따라감) | **사용자가 제어** (재생/일시정지/시크/배속) |
| 윈도우 종료점 | 알 수 없음 (라이브 엣지가 계속 움직임) | **세션 `date_start` / `date_end`로 사전 확정** |
| 폴링 cadence | 시계 기반 (균등) | 재생 위치 기반 (선행 fetch) |
| 중복 방지 수단 | `date>=cursor` (커서 진행) | **반-개구간 `[T, T+W)`** |
| 인증 | 라이브 데이터는 유료 권장 | **무료/익명** |
| 다중 사용자 부하 분산 | 백엔드 1개가 OpenF1 폴 → N 사용자에게 동일 push | **본 MVP에서는 해당 없음** (동접 1명, 브라우저 메모리 캐시만). 외부 공개 시점에 백엔드 윈도우 캐시 도입 |
| 흡수해야 할 위험 | 라이브 지연/끊김/429 | 거의 없음 (세션 미러 끝남) |

### 1.3 요청 수 차이의 출처 (라이브와 재생)

라이브와 재생은 **받는 데이터 양이 동일**(`location` 기준 분당 ~4,440 records)하지만 요청 횟수는 라이브가 ~6배 많다. 이는 OpenF1 한계가 아니라 **배치 전략의 차이**다.

| 모드 | 1회 응답 시간 폭 | 1회 응답 레코드 | 분당 요청 | 분당 레코드 |
|---|---|---|---|---|
| 라이브 (10s 폴) | 10s | ~740 | **6** | 4,440 |
| 재생 (60s 윈도우) | 60s | ~4,440 | **1** | 4,440 |

**메커니즘:**
- **라이브**는 아직 생성되지 않은 미래 데이터를 받을 수 없다 → **폴 간격 = 1회 응답 시간 폭**. 그 폴 간격의 상한은 표시 버퍼 깊이(30s)에 의해 결정되며, 안전하게 `buffer / 3 ≈ 10s`. 따라서 작은 응답을 자주 받는 구조가 강제된다.
- **재생**은 미래 데이터가 이미 존재한다 → **윈도우 크기를 자유롭게 키워 한 번에 받을 수 있다.** 재생 시계가 그 윈도우를 다 소진하기 전에 다음 윈도우만 prefetch하면 충분.

**함의:**
- 재생에서 윈도우 크기를 임의로 줄이면(예: 10s 윈도우) 단지 요청 수만 6배가 될 뿐 얻는 이점이 없다.
- §3.2의 60s 선택은 응답 크기 / 시크 비용 / 캐시 입자 간 균형의 결과이지, 라이브 폴 cadence를 흉내 낸 결과가 아니다.
- 반대로 라이브의 요청 수는 (표시 지연 ↑) ↔ (요청 ↓) 트레이드오프로 더 줄일 여지가 있으나 "라이브감"을 양보해야 한다 — 본 시스템은 라이브 30s 지연을 유지하므로 6 req/min를 채택.

## 2. 핵심 기법: 반-개구간 윈도우 폴링

### 2.1 OpenF1 쿼리 지원 사실 확인
- OpenF1는 동일 필드에 `>=`와 `<`(혹은 `>`/`<=`)를 함께 걸 수 있고, AND로 결합된다.
- 타임스탬프는 UTC ISO 8601, 마이크로초 정밀도 (`2024-03-02T15:00:00.123456+00:00`).
- 동일 차량의 같은 종류 레코드가 동일 `date`를 갖는 일은 사실상 없다.

### 2.2 채택 규약: **좌-닫힘 / 우-열림** `[T1, T2)`
```
GET /v1/location?session_key=9472
   &date>=2024-03-02T15:00:00Z
   &date<2024-03-02T15:01:00Z
```
- 각 레코드는 정확히 한 윈도우에 속한다 → 인접 윈도우 합치면 정확히 원본.
- 클라이언트 측 dedup 코드 불필요.
- 경계 시각 자체에 떨어진 레코드는 다음 윈도우의 첫 레코드가 된다 (직관과 일치).

### 2.3 비채택: 양-닫힘 `[T1, T2]`
- `date>=T1 & date<=T2`는 다음 윈도우 `date>=T2`와 시각 `T2` 레코드가 중복.
- dedup 코드 필요 + 결정성 떨어짐 → 사용하지 않는다.

### 2.4 비채택: `>` 만 사용 (`>T1 & <=T2`)
- 우-닫힘이라 마지막 레코드가 직관과 어긋남. 좌-닫힘/우-열림이 표준 시계열 규약(예: `pandas`, `numpy`, SQL `BETWEEN ... AND` 대신 `>= AND <`)과 일치하므로 그쪽을 채택.

## 3. 엔드포인트별 fetch 전략

세션 길이 약 2시간(120분), 풀 그리드(20대) 가정.

### 3.1 풀세션 1회 (희박/이벤트성)
세션 전체를 한 번에 가져와 메모리/캐시에 보유. `date` 필터 불필요 (`session_key`만으로 충분).

| 엔드포인트 | 풀세션 응답 크기 (대략) | 비고 |
|---|---|---|
| `laps` | ~1,600 records | 드라이버 × 랩 |
| `weather` | ~120 records | 분당 1 |
| `race_control` | ~30~150 records | 이벤트성 |
| `pit` | ~30~80 records | 핏 이벤트 |
| `stints` | ~40~80 records | 스틴트 |
| `session_result` | 20 records | 최종 결과 |

→ **6 requests로 세션 전체 정적 데이터 확보**. 메모리에서 시간순 인덱싱 후 재생 시계 진행에 맞춰 노출.

### 3.2 윈도우 분할 fetch (조밀 시계열)

| 엔드포인트 | 데이터 밀도 | 채택 윈도우 | 1윈도우 응답 (대략) | 2시간 세션 총 윈도우 수 |
|---|---|---|---|---|
| `location` | ~3.7 Hz × 20대 = ~74 rec/s | **60s** | ~4,440 records | 120 |
| `position` | 이벤트 (변화 시) | **60s** | ~10~50 records | 120 |
| `intervals` | ~0.25 Hz × 20대 = ~5 rec/s | **60s** | ~300 records | 120 |

- 60s 윈도우는 응답 크기 균형이 좋음. JSON 파싱 시간 < 200 ms 수준 예상.
- 윈도우를 더 크게(예: 300s) 해도 무방하나 시크 후 부분 폐기 비용이 커짐.
- 윈도우를 더 작게(예: 10s) 하면 요청 수가 6배가 되어 다중 사용자 동시 재생 시 캐시 효율 ↓.

### 3.3 윈도우 시간 정렬 (Snap)
- 모든 윈도우는 **세션 시작 시각(`sessions.date_start`)을 기준으로 60s 그리드에 snap**한다.
  - 예: `date_start = 15:00:23Z` 라면 윈도우 경계는 `15:00:23`, `15:01:23`, `15:02:23`, ...
- 모든 사용자가 같은 그리드를 사용하므로 윈도우 단위 캐시가 사용자 간 공유된다.
- 시크/배속 변경 시에도 그리드가 그대로라 캐시 적중률 유지.

### 3.4 세션 경계 처리
- 세션 데이터는 `date_start`보다 약간 앞(아웃랩 시작), `date_end`보다 약간 뒤(쿨다운 랩)까지 존재할 수 있음.
- 첫/마지막 윈도우는 그리드 경계 외 데이터를 놓치지 않도록 **각각 60s 패딩**을 둠.
  - 첫 윈도우: `[date_start − 60s, date_start − 60s + 60s)`
  - 마지막 윈도우: `[date_end, date_end + 60s)`
- 빈 응답(`[]`)이어도 비용은 적으므로 안전.

## 4. 재생 시계와 룩어헤드 버퍼

### 4.1 재생 시계 정의
- `playback_clock` — 사용자가 제어. 단위는 UTC.
- 재생 속도 `speed` (기본 1.0). 일시정지 시 0.
- 매 프레임마다 `playback_clock += dt × speed`.

### 4.2 룩어헤드 버퍼 규칙
- 항상 `playback_clock + 60s × speed`까지의 데이터를 메모리에 유지.
- `playback_clock`이 현재 보유 윈도우의 **중간점**을 지나면 다음 윈도우를 비동기 prefetch.
- 1x 재생에서 60s 룩어헤드 = 1 windowahead = 매 분 ~3 requests (`location` + `position` + `intervals`).
- 4x 재생에서는 룩어헤드를 **240s = 4 windows**로 자동 확장. 매 분 ~12 requests.

### 4.3 시크 (Scrub)
- 사용자가 시크 → `playback_clock`을 새 위치로 점프.
- 신규 위치를 포함하는 윈도우가 캐시에 없으면 즉시 fetch.
- 이전 룩어헤드 데이터는 폐기 (단, 윈도우 단위 캐시는 보존 — 사용자가 되돌아오면 재사용).
- 시크 직후 첫 프레임 표시까지 약 200~500 ms 지연 가능 → UI에 짧은 로딩 인디케이터.

### 4.4 배속 변경
- 룩어헤드 = `60s × speed`로 즉시 재계산.
- 배속 ↑ 시 추가 윈도우 prefetch (한 번에 여러 윈도우 병렬).
- 배속 ↓ 시 잉여 윈도우는 그대로 캐시에 유지.

### 4.5 재생 시계 vs 라이브 모드 비교
| | 라이브 모드 | 재생 모드 |
|---|---|---|
| 시계 원천 | `newest_received_date − 30s` | 사용자 제어 |
| 흐름 | 데이터에 의해 끌려감 | 사용자가 끌고 감 |
| 끊김 시 | UI: "데이터 지연 중", 시계 천천히 진행 | UI: "버퍼링", 시계 정지 |
| 버퍼 깊이 | 30s (다운/지터 흡수용) | 60s × speed (smooth prefetch용) |

## 5. 캐시 모델 — **본 MVP: 브라우저 세션 메모리**

본 MVP는 동접 1명 가정이므로 다중 사용자 LRU·request coalescing이 불필요하다. 같은 세션을 재생하는 동안만 윈도우 응답을 메모리에 보존해, 시크/배속 변경 시 이미 받은 윈도우를 재폴 없이 즉시 사용한다.

### 5.1 캐시 키와 저장소
```
cache_key = (endpoint, session_key, window_start_iso)
```
- 저장소: **JS `Map`** (메모리). 페이지/탭이 닫히면 사라짐.
- TTL 없음 (historical은 immutable). 페이지가 열려 있는 동안 유지.

### 5.2 캐시 동작
- 윈도우 fetch 직전 `Map.get(cache_key)` 조회 → 있으면 즉시 반환, 없으면 OpenF1 호출 후 `Map.set`.
- 시크: 새 위치의 윈도우가 캐시에 있으면 OpenF1 호출 0회. 없으면 1회 fetch (~500ms).
- 배속 변경: 룩어헤드 윈도우를 prefetch (캐시 적중이면 비용 0).
- in-flight 중복 호출 방지(동일 cache_key를 동시에 두 번 요청)는 동접 1명이라도 시크/룩어헤드 prefetch가 겹칠 때 의미가 있어 **단일 in-flight promise map**으로 합친다 (수 줄 코드).

### 5.3 캐시 한계와 후속 옵션
- **재방문(탭을 닫았다가 다시 열기) 시 캐시 미적중** — 같은 세션을 다시 폴해야 한다. 무료 한도 안이라 비용 문제는 아니지만 첫 표시까지 ~1~2초의 워밍업이 매번 발생.
- **오프라인 재생 불가** — 네트워크가 없으면 진입조차 불가.
- 위 두 한계가 사용자 불편으로 드러나면 후속으로 IndexedDB 영구 캐시 도입을 검토 (현 MVP 스코프 밖). `idb` 라이브러리(~1KB) + 캐시 무효화 정책이 추가됨.

### 5.4 핫 세션 사전 적재 (선택, 본 MVP에서는 GitHub Actions 빌드 타임에 정적 JSON으로)
사용자가 자주 보는 세션(예: 최신 race)은 GitHub Actions가 빌드 타임에 풀세션을 fetch해 `src/data/preloaded-sessions/{session_key}.json` 으로 commit해 둘 수 있다. 진입 시 정적 import로 즉시 사용 → OpenF1 호출 0회. 본 MVP에서는 옵션으로만 두고 첫 구현에는 포함하지 않는다.

> **향후 확장: 백엔드 다중 사용자 캐시.** 외부 공개 시점에는 백엔드 LRU + request coalescing이 다시 필요해진다. 그 단계의 설계는 본 문서의 이전 판(`git log`)을 참조할 것.

## 6. 요청 수 계산

### 6.1 단일 사용자 1x 재생, 2시간 세션
| 분류 | 요청 수 |
|---|---|
| 정적/희박 풀세션 (laps/weather/race_control/pit/stints/session_result) | 6 |
| `location` 윈도우 120개 | 120 |
| `position` 윈도우 120개 | 120 |
| `intervals` 윈도우 120개 | 120 |
| **합계** | **366 requests / 2시간** |
| **분당 평균** | **~3.0 req/min** |

→ 무료 30 req/min 한도의 **10%**. 매우 여유.

### 6.2 단일 사용자 4x 재생
- 같은 데이터를 4배 빠르게 소비 → 분당 평균 ~12 req/min.
- 무료 한도 내. 한도의 40%.

### 6.3 다중 사용자 (향후 확장 참고)
본 MVP는 동접 1명이라 해당 없음. 외부 공개 시점에 백엔드 LRU 캐시 도입 후 적중률 `h`에 따라:
```
OpenF1 부하 = (사용자 수) × (사용자당 req/min) × (1 − h)
```
- 100명 동일 세션, 적중률 95% → ~15 req/min (free tier 안)
- 100명 서로 다른 세션, 적중률 0% → 300 req/min (free 초과, 스폰서 또는 큐잉)

---

## 7. 무료 vs 유료 차이

### 7.1 무료 (REST)
- Historical 데이터는 **모두 무료/익명**으로 접근 가능.
- 본 전략은 무료 플랜만으로 완성됨.
- 단일/소수 사용자: 무한정 가능.
- 다수 사용자 + 다양한 세션: 캐시 적중률 따라 free tier 초과 가능 → 스폰서 전환 또는 요청 큐잉.

### 7.2 유료 (스폰서)
- 한도 60 req/min로 두 배. 다중 사용자 + 다양한 세션 시나리오에서 여유.
- MQTT는 라이브 푸시 채널이라 재생에는 직접적 이점 없음.
- 유료의 진짜 가치는 라이브 모드. 재생 전용이라면 스폰서가 강제되지 않음.

## 8. 코드 설계 가이드

### 8.1 브라우저 컴포넌트 (본 MVP)

본 MVP는 백엔드가 없으므로 모든 컴포넌트가 사용자 브라우저에서 동작한다.

1. **SessionMetadataLoader** — 재생 시작 시 `sessions`/`meetings`/`drivers`/희박 풀세션 6개 fetch.
2. **WindowFetcher** — 반-개구간 fetch 단일 책임. 60s 그리드에 snap. 응답을 메모리 캐시에 저장.
3. **WindowCache** — `(endpoint, session_key, window_start)` 키의 **JS `Map`** (단일 사용자 메모리 캐시). in-flight 합치기.
4. **PlaybackController** — `playback_clock`, `speed`, 룩어헤드 정책. 다음에 필요한 윈도우 목록을 계산해 Fetcher에게 위임.
5. **`ReplayDataSource` 어댑터** — 위 4종을 묶어 라이브맵·대시보드가 사용할 공통 `DataSource` 인터페이스로 노출.

### 8.2 라이브 ↔ 재생 공통 추상
- `DataSource` 인터페이스 — 라이브에선 REST 폴러 어댑터, 재생에선 WindowFetcher.
- `EventStream` — 시간순 정렬된 레코드 시퀀스. 라이브와 재생 모두 동일 인터페이스.
- 클라이언트의 보간기/렌더 루프는 두 모드를 구분하지 않는다 (재생 시계와 표시 시계가 같다는 점만 다름).

### 8.3 UI 컴포넌트
- 라이브와 동일한 렌더 루프 재사용.
- 추가 컨트롤: 재생/일시정지/시크 바/배속(0.5x/1x/2x/4x).
- 보간기는 라이브와 같은 차량별 독립 시계 방식.

> **향후 확장: 백엔드 도입 시.** WindowCache를 백엔드 LRU + Redis로 이전하고 ClientStreamer(WebSocket push 또는 백엔드 REST 프록시) 추가. 브라우저는 `ReplayDataSource` 어댑터만 백엔드 호출로 교체.

## 9. 운영 지표 — 본 MVP는 브라우저 dev console로 단순화

본 MVP는 백엔드 모니터링 인프라가 없으므로 다음 지표를 브라우저 콘솔/in-page debug 패널로 노출한다 (운영 알림 채널 없음).

| 지표 | 정상 범위 | 동작 |
|---|---|---|
| 평균 윈도우 응답 시간 (`location`, 60s) | < 500 ms | > 2s 지속 시 UI에 "네트워크 지연" 안내 |
| 시크 후 첫 프레임 지연 | < 500 ms | > 2s 시 시크 바에 로딩 인디케이터 유지 |
| OpenF1 호출 분당 합계 | < 30 | 한도 임박 시 콘솔 경고 (정상은 ~3 req/min) |
| 429 응답 | 0회 | 1회라도 발생 시 즉시 콘솔 경고 + 자동 백오프 |
| 캐시 적중률 (메모리) | — | 디버그 패널 표기 (정보용, 알림 없음) |

> **향후 확장 (백엔드 도입 시):** 캐시 적중률·활성 세션 수·OpenF1 호출 합계를 Prometheus/Grafana 등으로 메트릭화하고 알림 임계 부활.

## 10. 주의 사항

1. **`intervals` historical 비신뢰성** — OpenF1 문서가 `intervals`를 "live only"로 기술. 재생 모드에서 빈 응답 가능. 대응:
   - **(권장)** UI에서 갭 표시를 "데이터 없음"으로 우아하게 처리.
   - **(대안)** 풀세션의 `position` + `laps`로부터 랩 단위 갭을 산출 (단, 라이브의 ~4s 해상도가 아니라 랩 단위로 snap). 라이브 모드에서는 이 derive를 채택하지 않았지만, 재생 모드에서는 한 번 사전 계산해 두면 안정적으로 사용 가능 — 채택 여부는 §11 결정 사항.
2. **`pit.stop_duration` null (2024 US GP 이전)** — 2024-10-20 이전 세션 재생 시 정차 시간 표기 누락. UI는 `lane_duration`만 표시하도록 fallback.
3. **2024 Sprint Qualifying 일부 랩/섹터 누락** — 업스트림 피드 결손. 재생 시 해당 구간 빈 셀로 표시.
4. **Miami 2026 라이브 결손이 historical에도 반영** — 라이브 수집이 실패한 세션은 재생 모드에서도 그대로 비어 있음. 세션 단위로 데이터 완전성 사전 확인 필요.
5. **`location` (0,0,0) 가라지 sentinel** — 라이브와 동일. 실주행 좌표로 간주하지 말 것.
6. **`intervals` 문자열 갭** (`"+1 LAP"`) — 라이브와 동일. 숫자 연산 전 타입 체크.
7. **2023 이전은 공개 API에서 일체 제공되지 않음** — 자가 호스팅 ingestor가 필요. 본 전략은 2023+ 세션만 대상.
8. **세션 식별 UX** — 사용자가 재생할 세션을 선택할 때 `meetings`/`sessions` 카탈로그를 미리 캐시해 두면 별도 API 호출 없이 즉시 선택 가능.
9. **윈도우 크기를 라이브 폴 cadence와 혼동하지 말 것** — 재생의 60s 윈도우는 응답 크기 / 시크 비용 / 캐시 입자 균형에서 나온 값이며, 라이브가 10s 폴인 것과는 무관. 라이브와 같은 cadence(예: 10s 윈도우)로 잡으면 동일한 데이터를 얻기 위해 요청 수가 6배가 된다. 재생에서는 미래 데이터가 이미 존재하므로 **큰 윈도우 + 드문 호출이 항상 효율적**이며, 라이브의 폴링 패턴을 재사용하지 말 것. (배경 §1.3)
10. **윈도우 크기를 과도하게 키우지도 말 것** — 반대로 5분/10분 같은 큰 윈도우는 (a) 응답 파싱 시간 ↑, (b) 시크 시 부분만 필요해도 큰 윈도우를 통째로 받아 폐기, (c) 캐시 입자가 거칠어 사용자 간 캐시 적중률 ↓ 등의 비용이 있음. 60s가 본 운영의 균형점.

## 11. 미해결 / 후속 결정 필요 항목

1. **`intervals` derive 채택 여부** — 재생 모드에서 라이브 모드와 다르게 한 번 사전 계산해 두는 것이 안전하다는 판단이 가능. 채택 시 `position` + `laps` 기반 derive 모듈을 어디(백엔드/클라이언트)에서 돌릴지.
2. **세션 카탈로그 갱신 주기** — `meetings`/`sessions`는 매일 자정 UTC 갱신. 캐시 무효화 정책 (24h TTL or webhook 부재로 폴링).
3. **핫 세션 사전 적재 트리거** — 어떤 세션을 자동 사전 적재할지 (최신 race, 사용자 선택 빈도 상위 N개 등).
4. **풀세션 사전 적재가 단일 사용자에게도 유리한 시점** — 모바일/저대역 사용자에게는 전체 prefetch가 더 부드러울 수 있음.
5. **시크 시 UX** — 윈도우 fetch 동안 보여줄 UI (마지막 프레임 정지 vs 검은 화면 vs 스피너).
6. **라이브 → 재생 끊김 없는 전환** — 사용자가 라이브 시청 중 30초 전으로 되감기 누를 때, 라이브 30s 버퍼와 재생 윈도우 캐시 사이의 일관성.
